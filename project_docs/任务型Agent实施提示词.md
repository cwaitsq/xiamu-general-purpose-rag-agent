# 任务型Agent实施提示词

下面这段提示词用于驱动后续 AI 或协作开发者继续把本仓库往“企业级 Agent 应用”方向推进。

## 建议直接使用的提示词

```text
你现在在维护一个企业级 Agent 应用仓库，而不是一个简单的聊天 Demo。请在不破坏现有前端页面的前提下，优先增强后端、工作流、日志、评测和任务闭环能力。

项目目标：
1. 保持现有 RAG 问答、知识整理、知识入库链路可运行。
2. 强化任务型 Agent 能力，让系统不仅能回答问题，还能完成任务。
3. 输出结果必须可观测、可评测、可复盘。

你在这个仓库里的工作原则：
1. 先读 README、project_docs、backend_service、gateway、tools、n8n工作流。
2. 不要随意改前端页面结构，除非明确要求。
3. 优先把复杂逻辑落到后端服务、脚本和工作流生成器里。
4. 每做完一个能力点，都补测试或评测脚本。
5. 文档必须同步更新，让后续维护者可以直接接手。

优先实现顺序：
1. 任务型 Agent：增加订单查询、库存核验、建工单、人工转交等工具调用闭环。
2. 可观测性：记录步骤日志、工具调用日志、失败原因、耗时、任务状态。
3. 评测：提供固定 benchmark，用准确率、拒答率、幻觉率、转人工率、平均耗时衡量结果。
4. 工作流：在 n8n 中增加可直接演示的 webhook 工作流，打通请求 -> Agent -> 结果返回。
5. 文档：给出架构说明、接口说明、演示方式、面试讲解重点。

输出要求：
1. 代码必须可运行。
2. 每个新增接口都要有明确返回结构。
3. 每次提交后给出验证命令和验证结果。
4. 如果存在“演示型实现”和“生产化建议”的差异，必须明确写出来。

验收标准：
1. 至少有一个真实闭环任务，不只是问答。
2. 至少有一条新的 n8n 工作流演示任务型 Agent。
3. 至少有一份可执行评测脚本。
4. 至少有一套可读的日志 / trace 数据结构。
5. 文档里能讲清楚边界、取舍、风险和后续优化方向。
```

## 为什么这段提示词有效

- 它把目标从“做个 AI 功能”改成了“维护一个企业级 Agent 系统”。
- 它明确限制了前端改动范围，减少无效返工。
- 它要求每次能力增强都补测试、评测和文档，能逼着输出结构化结果。
- 它把面试展示最重要的几件事直接写进了验收标准。

## 推荐你继续往下压的方向

- 把当前任务型 Agent 的工具注册机制再抽象一层，做成可插拔工具目录。
- 给任务执行链路增加更细的权限校验和审计字段。
- 把基准评测拆成离线评测和线上真实运行指标两条线。
- 把“订单/库存/工单”再扩成更多企业内部 API 场景。
